# signal-trader 无 writer live 收口计划

## 目标

把当前 `apps/signal-trader` 的 live 启动方式，收敛成“app 默认入口直接支持基于 VEX 的 live + 环境变量驱动启动”，并去掉对单独 `okx-order-writer` 这个软件组件的默认依赖：`signal-trader` 仍可依赖 SQL `"order"` 表作为 closed order history 证据源，但该表的数据生产责任下沉到 VEX/叶子节点或外部基础设施，不再由 `signal-trader` package 默认捆绑一个 writer 进程。

## 现状判断

- `apps/signal-trader/src/index.ts` 当前是默认安全入口，只启动 core + SQL repositories，不自动接入任何 live provider。
- `apps/signal-trader/dev/bootstrap-live-app.js` 当前承担的是 OKX local live 装配职责：secret resolver、live venue、observer provider、capability registry、service policy 都写在这里。
- `apps/signal-trader/dev/run-local-live-stack.sh` 同时承担环境编排（compose、migration、pid/log）与启动入口职责，导致用户感知成“重要配置都写在脚本里”。
- 这种分层本身是为了 `secure-by-default` 和 `fail-close`，不是因为 app 没有主体；但按用户预期，app 入口就应该负责把 live 所需依赖从 env 装配好，而不是把核心启动逻辑留在 dev 脚本里。
- 用户进一步澄清：`signal-trader` 不是替代 VEX 的执行层，而是 VEX 的上游；secret 处理与真实下单应由 VEX 负责。
- 因此，当前 app 里的“直接 resolve secret / 直接管理 live venue / 自己声明 direct-vendor capability”的假设需要重审；live 入口应首先围绕 VEX 作为下游边界来设计。
- 当前默认 live backend 仍依赖 `closed order history` 能力；在本地 OKX profile 里，这个能力由 `apps/vendor-okx/src/order.ts` 持续写入 SQL `"order"` 表，因此 compose 中会出现单独的 `okx-order-writer`。用户现已明确：可以依赖 SQL `"order"` 表本身，但不接受从软件交付层面默认依赖一个特定 writer 进程；归档审计责任应下沉到 VEX/叶子节点。

## 推荐方案

1. 改造 `apps/signal-trader/src/index.ts`，使其默认入口直接支持基于 env 的 paper/live 启动，而不是只起 core。
2. 将 live 语义改成“VEX 作为下游执行/路由层”的官方模型，而不是 app 自己直连 vendor：
   - `signal-trader` 负责 signal/effect/runtime/fail-close
   - VEX 负责 secret、真实下单、真实撤单
3. 保留 app 对 SQL `"order"` 历史证据源的读取能力，但调整“依赖边界”的表达方式：`signal-trader` 依赖的是 SQL `"order"` 表契约，而不是某个具体 writer 进程。
4. 调整 app 默认 live 契约：把 `closed order history` 从“默认打包职责”中剥离，但继续作为已支持的证据源；默认 live 的最小运行前提仍要求：
   - `SubmitOrder`
   - `CancelOrder`
   - `QueryPendingOrders`
   - `QueryAccountInfo`
   - route proof / account-bound 验证
5. 在 app 内部 env 装配层（例如 `src/bootstrap-from-env.ts`）中，由默认入口自动构造：
   - 基于 account-bound 服务的 VEX `liveVenue`（`SubmitOrder` / `CancelOrder`）
   - 基于 account-bound 服务的 `observerProvider`（`QueryPendingOrders` / `QueryAccountInfo`），并继续可选读取 SQL `"order"` 表
   - 面向 VEX 路径的 `liveCapabilityRegistry`
   - `servicePolicy`
6. 重新审视 runtime live 配置契约：
   - 默认 VEX live 路径以 `account_id` 为核心路由键，而不是要求 app 直接持有/解析交易所 secret
   - account-bound 路径应弱化或移除“live 必须提供 `secret_ref`”的校验，并同步改写 validation、README、seed 脚本
7. 重写 observer / fail-close 语义：
   - `signal-trader` 继续优先消费 SQL `"order"` closed history + `QueryPendingOrders` + `QueryAccountInfo`
   - 若订单已离开 `QueryPendingOrders`，但 SQL `"order"` 中也没有终态证据，则直接 fail-close 到 `audit_only`，要求人工 backfill 或下游补终态证据
   - 若 open orders / account snapshot / route proof 缺失或漂移，也继续锁 runtime
   - 审计日志明确区分“依赖的 SQL order history 不可用”和“软件交付不再默认捆绑 writer”
8. 收敛 `apps/signal-trader/dev/docker-compose.live-okx.yml` 与 `run-local-live-stack.sh`：
   - 从 signal-trader 的默认交付里移除 `okx-order-writer`
   - 默认 compose 不再追求开箱即全链路；compose / runbook 只声明需要一个已由 VEX/叶子节点维护的 SQL `"order"` 表数据源
   - `signal-trader` 继续作为 compose 服务直接读取 env 启动
9. 更新 `README.md` / `GUIDE.md`，把启动模型改写为：
   - 默认入口：直接支持 paper/live，live 默认走 VEX 下游
   - 默认 live backend 仍可依赖 SQL order history 表，但不再默认捆绑 writer 进程
   - archive audit 数据生产由 VEX/叶子节点自行提供与演进

## 边界与风险

- 不能把所有 live 差异“假装成纯 env 差异”；即便移除默认 writer 打包，SQL `"order"`、open orders、account snapshot、route proof 的来源仍需明确且可审计。
- 若用户要求 `signal-trader` 不碰真实 secret，则 `resolveSecretRef` 不应再出现在默认 VEX live 路径里，或至少不再承担“解析出真实 credential”的语义。
- `liveCapabilityRegistry` 不能被绕过；它仍是 support matrix / audit / fail-close 的真源。
- 让默认入口直接支持 live 后，必须重新定义 `secure-by-default`：默认没有显式 VEX/live 所需 env 时不得隐式起 live，而是安全失败或回落到明确模式。
- 如果仅从 compose 里删 writer、却不同时把“SQL `"order"` 由谁负责生产”改成外部前提并更新文档/校验，用户会误以为这条证据链消失了；实际上 app 仍会继续读取 SQL `"order"`，只是生产责任不再由 signal-trader 默认打包承担。

## 验证

1. `apps/signal-trader/dev/docker-compose.live-okx.yml` 与 `run-local-live-stack.sh` 不再包含/依赖 `okx-order-writer`，但 runbook 明确要求 SQL `"order"` 表应由 VEX/叶子节点或外部基础设施维护。
2. 默认 live 仍会消费 SQL `"order"`；若订单离开 `QueryPendingOrders` 且 SQL `"order"` 无终态证据，runtime 继续 `audit_only` fail-close。
3. `SignalTrader/ListLiveCapabilities`、审计日志与文档都能准确表达“依赖 SQL order history 表，但不依赖特定 writer 软件组件”。
4. 缺少 `QueryPendingOrders`、`QueryAccountInfo`、route proof、submit/cancel 能力时，runtime 仍然 fail-close。
5. 回归验证包括：`node --check`、`bash -n`、`docker compose config`、targeted build/test，以及一条“无内置 writer，但外部 SQL order history 前提已满足”的 account-bound VEX live happy-path smoke。
