## 非阻塞建议

- **建议 1：首版可进一步收敛安全策略复杂度**

  - 质疑点：`https-only + localhost 例外`、`30s timeout`、`1 MiB body limit` 同时作为 MUST，会让“一个轻量 util”在首版承担较多策略责任。
  - 最小化复杂度建议：保持 RFC 当前写法可实现，但实现时将这三项集中在单一常量区并明确“后续可参数化”，避免散落逻辑导致维护成本上升。

- **建议 2：明确 body-limit 与 NDJSON 流式解析的实现口径**

  - 质疑点：若先整包读取再限流，内存行为与“资源保护”目标不一致；若流式累计字节数则更符合目标。
  - 最小化复杂度建议：文档补一句“按流累计字节并在超限时提前失败”，避免实现分叉。

- **建议 3：错误对象附加字段做最小白名单**
  - 质疑点：RFC 要求附带上下文，但未约束字段范围，易出现实现者附带过多细节。
  - 最小化复杂度建议：在 RFC 固定最小字段（如 `status/statusText`、`host`、`method`），其余不要求。

## 修复指导

1. 保持当前主流程不变：参数校验 → `POST /request` → NDJSON 逐行解析 → 首个 `res` 返回 → 无 `res` 返回 `NO_RESPONSE`。
2. 将 timeout/body-limit/transport-policy 的实现集中到一个内部配置块，减少分散判断。
3. 在测试中优先锁定最小可验行为（成功、HTTP 错误、解析错误、无 res、参数错误），扩展策略测试可作为增量。
