# Research Notes（会议结论映射）- 实盘交易库 heavy RFC

> 目标：仅基于本次会议已确认结论，收敛一版可评审、可实现、可回滚的 heavy 设计输入；本文件不引入 scope 外议题。

---

## 1. 输入边界与方法

- 输入来源：`task-brief.md` 与会议结论（已拍板/否决/TBD）。
- 工作边界：`stage=design-only`，仅文档产出，不改生产代码。
- 强制架构边界：单一 core lib，运行时无副作用（不做 DB/网络/消息/进程）。

## 2. 会议结论映射（逐条落位）

### 2.1 已基本拍板（冻结为 V1 默认）

1. 输入采用 push（不是 pull）
   - 设计落位：仅定义 `submit_signal` 命令入口，不定义 pull 轮询器。
2. V1 信号只支持离散三值 `{-1,0,1}`
   - 设计落位：`SignalValue = -1 | 0 | 1`，非法值直接拒绝并审计。
3. 执行层下单时直接挂止盈/止损
   - 设计落位：`OrderIntent` 必须同时包含主单与 TP/SL 参数。
4. 投资者状态按人分开记录，执行可合仓，归因需拆回
   - 设计落位：账本按 `investor_id` 独立；执行层可汇总为 `TotalPosition`，结算/审计按投资者反拆。
5. 信号日志与下单日志必须可审计
   - 设计落位：`SignalAccepted/Rejected` 与 `Order*` 事件强制落审计。
6. 核心模块做成可单测 library，外部副作用走抽象接口并支持 mock
   - 设计落位：核心 API 仅 `state + command -> next_state + events + effects`；副作用由 ports 执行。

### 2.2 明确否决（V1 禁止）

1. 不允许“预支未来 VC”追信号
   - 设计落位：若 `VC` 不足，返回 `INSUFFICIENT_VC`，记录审计并发通知 effect。
   - 禁止项：不支持负债式透支、不支持无限垫资模式。

### 2.3 有倾向但仍 TBD（进入 Open Questions）

1. 回测先用历史最大单笔浮亏，实盘是否转 P95/P99/窗口/遗忘待定
2. 交易账户倾向慢慢打钱，不一次性打满
3. “信号接收后至少打一手基本手”有偏好但属于 taste，需数据验证

### 2.4 为消除实现分叉而新增冻结（本轮 RFC 决策）

1. `stop_loss_ref_pct` 在 V1 按 `signal_key` 统一，不允许投资者覆盖
   - 原因：该项会直接改变 `PositionPlanner` 输入模型与审计口径，属于阻塞级契约，不宜延后到实现期。

## 3. 关键公式与语义冻结

### 3.1 仓位公式（V1 强约束）

- `Position_i = floor_to_lot( VC_i / (stop_loss_ref_pct * price_ref * contract_multiplier) )`
- `TotalPosition = Σ Position_i`
- 约束：`0 < stop_loss_ref_pct <= 1`，否则拒绝执行并审计。
- 单位约束：`price_ref` 为信号接收时价格快照，`contract_multiplier` 现货默认 `1`。

### 3.2 信号翻转默认语义

- V1 默认 `close_then_open`：先平现有反向仓，再开目标方向新仓。
- 理由：可降低并发反手导致的净曝险瞬时放大风险。

### 3.3 V1 明确不做

- 不做连续信号强度（如 `[-1,1]` 连续值）。
- 不走“传统基金份额制”作为默认路径。

## 4. 架构倾向与可执行性判断

- 采用分层状态机与命令/事件接口，保证可回放、可幂等、可单测。
- 核心库只输出 `planned_effects`，不直接执行 IO。
- 失败路径需与成功路径同等可审计，满足追责与复盘。

## 5. 风险盘点（仅与会议结论直接相关）

- VC 口径不一致风险：若信号层与投资者层参数冲突，仓位结果不可解释。
- 合仓归因风险：执行层合并后若无可逆分摊规则，会造成投资者结算争议。
- TP/SL 原子性风险：若下主单成功但保护单失败，需立即进入补偿策略并告警。
- 时序风险：重复或乱序信号若未做幂等与时序校验，可能触发重复开仓。

## 6. 研究结论（写入 RFC 的设计输入）

- 结论 1：V1 采用 push + 离散三值信号，冻结 `close_then_open`。
- 结论 2：仓位按 `Position_i` 公式逐投资者计算，执行可合并，结算与审计必须可反拆。
- 结论 3：VC 不足只允许“拒绝+记账+通知”，禁止预支未来 VC。
- 结论 4：核心实现必须是单一纯库，无任何运行时副作用；DB/网络/消息/进程全部外置。
- 结论 5：交付方式为 docs-only Draft PR；Merge 视为设计批准，merge 后评论 `continue` 才进入实现里程碑。
